# OpenEQ Testing Strategy & Protocols

This document details the end-to-end testing processes designed to guarantee the stability, audio quality, and real-time performance of **OpenEQ**. Testing DSP (Digital Signal Processing) applications requires specialized methodologies due to hardware dependencies and timing sensitivity.

---

## 🎯 1. Testing Philosophy & Layers

The OpenEQ test pyramid is divided into 4 main layers to ensure real-time thread safety and the mathematical accuracy of DSP filters:

```
          /\
         /  \     UI & System Integration Tests (Manual / UI Automation)
        /----\
       /      \    Performance & Real-Time Tests (Performance Benchmarks)
      /--------\
     /          \   Integration Tests (Audio Engine Lifecycle, Samplerate Match)
    /------------\
   /              \  Unit Tests (Pure DSP, vDSP FFT, Decibel Math)
  +----------------+
```

1.  **Unit Tests:** Pure mathematical and algorithmic verifications (Pure DSP, JSON serialization, etc.) that run entirely independently of Apple audio hardware or `AVAudioEngine` initialization.
2.  **Integration Tests:** Ensure that the audio graph nodes (`AVAudioEngine`, `AVAudioUnitEQ`, `AudioDeviceManager`) connect, disconnect, and handle sample-rate/channel mismatches correctly.
3.  **Performance Tests:** Verify real-time constraints, ensuring there are no heap allocations or lock contentions inside audio tap callbacks, and measure FFT loop latency.
4.  **UI & System Integration Tests:** Validate SwiftUI components, fader adjustment limits, and system aggregate device creation processes.

---

## 🧪 2. Unit Testing

Unit tests are written using the XCTest framework. DSP equations must be verified using small tolerance ranges (epsilon values).

### 2.1. Mathematical & DSP Conversion Tests
*   **Logarithmic Decibel Conversions:** Verify the limits and conversions of `dB = 20 * log10(Linear)` and `Linear = 10^(dB/20)`:
    *   `+24.0 dB` -> `~15.8489` linear gain amplification.
    *   `0.0 dB` -> `1.0` linear gain (neutral).
    *   `-24.0 dB` -> `~0.0631` linear gain attenuation.
    *   Protect against boundary values (like `-∞ dB` or negative values).
*   **vDSP Hanning Window Verification:** Verify that the coefficients generated by `vDSP_hann_window` are symmetrical and approach zero at the boundaries.

### 2.2. EQPreset & PresetStore Tests
*   **JSON Serialization/Deserialization:** Ensure custom `EQPreset` structures serialize and deserialize losslessly.
*   **Collision Prevention:** Confirm that imported presets acquire unique IDs (UUIDs) to prevent overwriting existing templates.
*   **Edge Cases:** Validate that empty-named presets are rejected and frequency/gain constraints remain bound to limits.

---

## 🔗 3. Integration Testing

Integration tests verify the status transitions and configurations of AVFoundation graphs and hardware interfaces.

### 3.1. AVAudioEngine Lifecycle
*   Ensure state transitions in `AudioEngineController` pass correctly through `.idle` -> `.preparing` -> `.ready` -> `.playing` -> `.paused` -> `.stopped`.
*   Verify that passing malformed files or unreadable URLs transitions the controller to `.failed(error)` gracefully without deadlocking.

### 3.2. Sample Rate & Channel Adaptations
*   Consecutively load files with varying sample rates (44.1 kHz, 48 kHz, 96 kHz) and channel configurations (Mono, Stereo) to verify that the output format updates dynamically.

### 3.3. Core Audio Aggregate Device (macOS 14.2+)
*   Ensure that aggregate device creation (`AudioHardwareCreateAggregateDevice`) and deletion operate without returning OSStatus errors.
*   Verify that `SystemAudioEQEngine` catches and adapts when the system default audio output changes.

---

## ⚡ 4. Performance & Real-Time (Real-Time Thread) Testing

The most critical aspect of audio applications is preventing audio dropouts (glitches) on the real-time thread.

### 4.1. Real-Time Thread Constraints (Profiling with XCTest)
*   **Zero Heap Allocations:** Under no circumstances should memory allocations (`malloc`, Swift array resizing, string concatenations) occur inside tap callbacks or `SystemAudioEQEngine` IOProc callbacks.
*   **No Locks:** Avoid using mutexes or dispatching synchronously to GCD queues within processing callbacks. Use lock-free ring buffers or atomic variables for thread communications.
*   **FFT Performance Benchmarks:** Measure that a 1024-point vDSP FFT conversion and spectrum binning completes in under 1 ms (preferably < 0.1 ms) using `self.measure`.

```swift
func testFFTPerformance() {
    let analyzer = SpectrumAnalyzer()
    let dummyBuffer = createDummyStereoPCMBuffer(sampleRate: 48000, frameCount: 1024)
    
    self.measure {
        for _ in 0..<100 {
            _ = analyzer.analyze(buffer: dummyBuffer)
        }
    }
}
```

## 🖥️ 5. UI and Visual Testing (SwiftUI & Snapshot)

Used to verify pixel-level drawings of UI controls, especially the dynamic Spectrum Analyzer and the EQ Curve.

### 5.1. Snapshot and Regression Tests
*   Ensure that peak level meters (`LevelMeterView`) and the clipping light (`ClippingIndicatorView`) render accurately at different amplitude levels using snapshot test assets.
*   Test graphic EQ fader constraints via Xcode UI Automation (`OpenEQUITests`).

## 📋 6. Manual Testing & Device Matrix

Subjective sound quality evaluations and device-specific audio behaviors are evaluated via manual test runs.

### 6.1. Device Test Matrix
Testing must be performed on the following hardware profiles before any minor or major release:
1.  **Built-in Mac Speakers:** MacBook Pro/Air internal drivers (verifying stereo/spatial configurations).
2.  **Analog Wired Headphones:** Output via the 3.5mm headphone jack.
3.  **Bluetooth Headphones:** Latency and connection reliability checks using AAC and SBC codecs.
4.  **External USB DACs / Audio Interfaces:** Verification of professional high-resolution sample rates (e.g., 96kHz, 192kHz) through external sound cards (Focusrite, Apogee, etc.).

### 6.2. Manual Test Case Matrix

| ID | Test Name | Initial State | Execution Steps | Expected Result |
| :--- | :--- | :--- | :--- | :--- |
| **MT-01** | Rapid Bypass Toggle | Audio playing, EQ active | Rapidly click the EQ Bypass button 5 times | No audio dropouts, pops, crackles, or application crashes. |
| **MT-02** | Volume Booster Max | Audio playing | Drag the volume booster slider to 200% | Volume should increase, and the clipping indicator should light up on peaks. |
| **MT-03** | Output Swap | System EQ running | Disconnect headphones (switch to speakers) | Audio should seamlessly failover to speakers, with EQ processing continuing. |
| **MT-04** | Corrupt Preset Import | App open | Import a malformed presets JSON file | The app must reject the file, show an error alert, and preserve existing presets. |

## 🚀 7. CI/CD Pipeline (GitHub Actions)

A template GitHub Actions workflow to build and run tests automatically on every PR or push to main/develop:

```yaml
name: OpenEQ CI Pipeline

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main, develop ]

jobs:
  build-and-test:
    runs-on: macos-14
    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Select Xcode Version
        run: sudo xcode-select -s /Applications/Xcode_15.4.app/Contents/Developer

      - name: Build Application
        run: |
          xcodebuild clean build \
            -project OpenEQ.xcodeproj \
            -scheme OpenEQ \
            -destination 'platform=macOS' \
            CODE_SIGNING_ALLOWED=NO

      - name: Run Unit & Integration Tests
        run: |
          xcodebuild test \
            -project OpenEQ.xcodeproj \
            -scheme OpenEQ \
            -destination 'platform=macOS' \
            CODE_SIGNING_ALLOWED=NO
```
